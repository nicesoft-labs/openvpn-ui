Сети и маршруты
Выделяет VPN-подсеть для клиентов (пул адресов «доверенной» зоны). Не должен пересекаться с сетями клиентов. Пример: server 10.0.70.0 255.255.255.0 → сервер 10.0.70.1, клиенты 10.0.70.2–254. Это не добавляет маршруты к вашим LAN — только создаёт адресное пространство VPN. Для tap можно использовать server-bridge
Добавляет маршрут на стороне сервера и делает подсеть достижимой через туннель. Клиентам НЕ рассылается. Полезно для дополнительной VPN-подсети/сегмента (напр., «гости»). Для site-to-site используйте с client-config-dir + iroute (привязка подсети к клиенту). Пример: route 10.0.71.0 255.255.255.0.
Рассылает всем клиентам маршрут к вашей LAN (split-tunnel). Нужен, чтобы клиенты видели домашнюю/офисную сеть через VPN. Сервер должен уметь доставлять ответы (статический обратный маршрут или NAT). Пример: push "route 10.0.60.0 255.255.255.0". Для «весь трафик через VPN» — push "redirect-gateway def1 bypass-dhcp".
Прокидывает основной DNS клиентам.
Резервный DNS.
Маршрут по умолчанию через VPN для всего клиентского трафика.
Каталог для клиентских статических настроек (ccd).
Сопоставление клиент ↔ IP (удобно для статических адресов).
